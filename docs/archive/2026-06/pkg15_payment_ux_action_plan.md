**Project:** Poatung (หมูนุ่น+เป๋าตุง) · SEVENDOG DEV
**Scope:** PKG-15 PromptPay / Slip Payment — Post-test polish round
**สถานะ:** แกนระบบผ่านแล้ว ✅ → รอบ polish ก่อนปล่อยใช้งานจริง
**วันที่:** 14 มิ.ย. 2026

## 0. TL;DR (อ่านอันนี้พอ ถ้าไม่มีเวลา)

แกน PKG-15 ผ่านหมดแล้ว (สร้าง QR / แชร์ / ลูกค้าจ่าย / ส่งสลิป / ตรวจสลิป / หน้าลิงก์โชว์สถานะถูกต้อง) เหลือ **3 ปัญหา UX หลังบ้าน** ที่ทำให้คนใช้สับสน

| # | ปัญหา | Priority | แก้ที่ |
|---|-------|----------|--------|
| 1 | จอ app เจ้าของร้านค้างหลังจ่ายเงินสำเร็จ | **P0** | State sync + Success state + Auto-polling |
| 2 | แชร์แล้วได้ข้อความ 2 ก้อน ดูซ้ำ/รก | **P1** | รวมเป็น share message เดียว |
| 3 | ปุ่ม "บันทึก QR" บนมือถือกด save ไม่ได้ | **P1** | ใช้ `navigator.share` เป็นหลัก + ลดชั้นปุ่ม save |

> **หลักการเดียวที่ต้องยึดทั้งงาน:** payment record บน backend คือ **single source of truth**
> ทั้งหน้า **ลิงก์ (ฝั่งคนจ่าย)** และหน้า **app (ฝั่งแม่ค้า)** ต้อง render จาก state เดียวกัน และเปลี่ยนหน้าเองเมื่อ status เปลี่ยน — ไม่ต้องรอให้ใครกดปุ่ม

## 2. ปัญหา + Root Cause

### P0 — จอ app เจ้าของร้านค้างหลังจ่ายสำเร็จ (สำคัญที่สุด)
**อาการ (จาก Image 3):** หัวจอขึ้น `✅ ชำระแล้ว` แล้ว แต่ด้านล่างยังโชว์ QR, รหัสอ้างอิง, และปุ่ม "บันทึก/แชร์ QR" อยู่ครบ → ลูกค้า/แม่ค้างงว่า "จบหรือยัง"

**Root cause:** หน้า app ไม่ได้ยึด backend status เป็น state จริง
- ตอนกด "ตรวจสอบสถานะ" backend ตอบ `PAID/VERIFIED` แต่ฝั่ง app แค่โชว์ toast/snackbar แล้วปล่อยจอเดิมไว้
- ไม่มี auto-polling → แม่ค้าต้องกดตรวจเอง (นี่คือ UX fail ตัวจริง)
- หน้าลิงก์ฝั่งคนจ่าย sync ถูก เพราะมัน fetch status จาก backend ตรง ๆ — แต่หน้า app ไม่ได้ทำแบบเดียวกัน

### P1 — แชร์ออกมา 2 ก้อน ดูซ้ำ
**อาการ (จาก Image 2):** กดแชร์ครั้งเดียว แต่ใน Messenger ขึ้น 2 bubble — ก้อนนึงมีลิงก์ อีกก้อนเป็นข้อความรายละเอียดที่ไม่มีลิงก์

**Root cause:** ฟังก์ชันแชร์ยิง payload ออกไป 2 ชุด (น่าจะแยก "ข้อความ" กับ "ลิงก์+preview" เป็น 2 call หรือ 2 block) → ในแชทเลยดูเหมือนส่งซ้ำ ลูกค้างงว่าต้องกดอันไหน

### P1 — ปุ่ม "บันทึก QR เป็นรูปภาพ" บนมือถือกด save ไม่ได้
**อาการ (จาก Image 1):** บน Chrome Desktop กด download ได้ แต่บนมือถือ / in-app browser (FB / LINE WebView) กดแล้วขึ้น error "refresh หน้าจอไม่ได้" save ไม่ได้

**Root cause:** ใช้ canvas → `toDataURL()` / `<a download>` ซึ่งมือถือและ in-app WebView ส่วนใหญ่ block หรือทำงานไม่สม่ำเสมอ → **เป็นข้อจำกัดของ platform ไม่ใช่บั๊ก** ดิ้นกับมันไปก็แพ้ ต้องเปลี่ยนกลไก

## 3. หลักการออกแบบ (ยึดไว้ทุกการตัดสินใจ)

1. **Backend status = single source of truth** — ทุกหน้า render จาก state เดียวกัน
2. **ระบบเปลี่ยนหน้าเอง ไม่บังคับให้คนกด** — auto-polling ทำงานแทนปุ่ม "ตรวจสอบสถานะ" (ปุ่มเหลือไว้เป็น fallback)
3. **ลิงก์คือศูนย์กลาง QR เป็นแค่เครื่องมือ** — ลิงก์ทำได้ครบ (โชว์ QR / บอกยอด / รับสลิป / ตรวจสถานะ / ขอบคุณ / ปิดรอบ) ส่วน QR แค่ช่องทางจ่าย
4. **รอบการจ่ายต้องมี "จุดจบ" ชัดเจน** — ทั้ง 2 ฝั่งต้องเห็นหน้า success เดียวกัน ไม่ค้างกลางทาง

## 4. State Machine มาตรฐาน (ใช้ทั้ง app และหน้าลิงก์)

```
PENDING ──(สลิปเข้า)──> SLIP_UPLOADED ──(ตรวจผ่าน)──> VERIFIED/PAID ──> CLOSED
   │                                                         
   └──(หมดเวลา)──> EXPIRED
```

| State | หน้าจอแสดง | ปุ่มที่โชว์ | Auto-polling |
|-------|-----------|-----------|--------------|
| **PENDING** | QR + ยอด + รหัสอ้างอิง | แชร์ลิงก์ · คัดลอกลิงก์ · เปิดกล่องสลิป | ✅ on (ทุก 5–8 วิ) |
| **SLIP_UPLOADED** | "ได้รับสลิปแล้ว กำลังตรวจสอบ" + spinner | (ห้ามสร้าง QR ซ้ำ) | ✅ on |
| **VERIFIED / PAID** | ✅ การ์ด success ใหญ่ | **สร้างรายการใหม่** (ปุ่มหลัก) · กลับหน้าหลัก (ปุ่มเล็ก) | ปิด |
| **EXPIRED** | "QR หมดอายุแล้ว" | สร้าง QR ใหม่ (ห้ามรับสลิปเพิ่มในรอบเดิม) | ปิด |

**กฎเหล็ก:** เมื่อ state = `VERIFIED/PAID` → **ซ่อน QR, ซ่อนกล่องรับสลิป, ซ่อนปุ่มแชร์, ซ่อนปุ่มสร้าง QR ในรอบเดิม** ทันที

## 5. แผนงานแยกตาม Priority

### 🔴 P0 — แก้จอค้างหลังจ่าย (ทำก่อน ปล่อยไม่ได้ถ้ายังไม่แก้)

**สิ่งที่ต้องทำ:**
1. ทำ `PaymentSuccessScreen` (หรือ success state ในหน้าเดิม) สำหรับฝั่ง **app เจ้าของร้าน** ให้เหมือนหน้าลิงก์
2. เมื่อ API status = `PAID/VERIFIED` → เปลี่ยน UI เป็น success **ทันที** (ไม่ใช่แค่ toast)
3. เพิ่ม **auto-polling เบา ๆ** ในหน้า QR: poll ทุก 5–8 วิ ช่วง 3 นาทีแรก เจอ paid → เปลี่ยนหน้าเอง
4. ปุ่ม "ตรวจสอบสถานะ" → ลดเป็น **fallback** (เผื่อ polling พลาด)
5. หน้า success ซ่อน QR / กล่องสลิป / ปุ่มแชร์ / ปุ่มสร้าง QR รอบเดิมทั้งหมด

**หน้า success ควรแสดง:**
```
✅ ชำระเงินเรียบร้อยแล้ว
รายการ: ตรวจสอบสลิปครั้งที่ 3
ยอดชำระ: ฿1.00
รหัสอ้างอิง: F1303412
เวลาตรวจสอบ: 08:50

ระบบตรวจสอบสลิปสำเร็จ — รอบนี้ปิดเรียบร้อย
กลับหน้าหลักใน 5 วินาที...

[ สร้างรายการใหม่ ]   กลับหน้าหลัก
```

**Acceptance Criteria:**
- [ ] ลูกค้าจ่าย + ส่งสลิป → ตรวจผ่าน → **หน้า app เปลี่ยนเป็น success เองภายใน ≤ 10 วิ** โดยแม่ค้าไม่ต้องกดอะไร
- [ ] เมื่อขึ้น success แล้ว ไม่มี QR / กล่องสลิป / ปุ่มแชร์ ของรอบเดิมโผล่อีก
- [ ] หน้า app กับหน้าลิงก์ แสดงสถานะตรงกันเสมอ

---

### 🟡 P1 — รวม share message เป็นก้อนเดียว

**สิ่งที่ต้องทำ:** ฟังก์ชันแชร์ยิง **payload เดียว** ที่มีข้อมูลครบ + ลิงก์ในก้อนเดียว (ดูตัวอย่างข้อ 7) ปล่อยให้ platform ดึง preview card จาก OG tags เอง — ห้ามส่งข้อความซ้ำเป็นก้อนที่สอง

**Acceptance Criteria:**
- [ ] กดแชร์ 1 ครั้ง → ขึ้นในแชท **1 ข้อความ** (อาจมี preview card 1 ใบ ตามปกติ ถือว่าโอเค) ไม่มี text block ซ้ำ
- [ ] ลิงก์อยู่ในข้อความนั้น กดแล้วเข้าหน้า pay ได้เลย

---

### 🟡 P1 — แก้ save QR บนมือถือ (เลิกสู้กับ browser)

**สิ่งที่ต้องทำ — เรียงลำดับตาม fallback:**
1. **ปุ่มหลัก = `navigator.share`** (Web Share API) — บนมือถือส่งได้ทั้งไฟล์ QR (`canvas.toBlob` → `File`) + ลิงก์ ผ่าน share sheet ของ OS เลย ลูกค้าเลือกเซฟ/ส่งต่อได้เอง
2. ถ้า `navigator.canShare({ files })` ไม่รองรับ (in-app WebView บางตัว) → **fallback: เปิดรูป QR เต็มจอ / แท็บใหม่** ให้ผู้ใช้แตะค้างที่รูปแล้วเลือก "บันทึกรูปภาพ" เอง (วิธีนี้ใช้ได้แทบทุกเครื่อง)
3. ปุ่มสำรอง **"คัดลอกลิงก์ชำระเงิน"** เผื่อทุกอย่างพัง
4. ปุ่ม "บันทึก QR เป็นรูปภาพ" → **ลดชั้นเป็นตัวรอง** ไม่ใช่ปุ่มหลักอีกต่อไป

**ข้อความใต้ปุ่ม:**
> หากบันทึกรูปไม่ได้ ให้กด "เปิด QR เต็มจอ" แล้วแตะค้างที่รูปเพื่อบันทึก หรือแชร์ลิงก์แทน

**Acceptance Criteria:**
- [ ] บนมือถือ (Chrome/Safari) กดแชร์แล้วเซฟ/ส่ง QR ได้ผ่าน share sheet
- [ ] ใน in-app WebView (FB/LINE) มี fallback ที่ใช้งานได้จริง (เปิดเต็มจอ + แตะค้างเซฟ)
- [ ] ไม่มี error "refresh ไม่ได้" โผล่อีก

---

### 🟢 P2 — robustness (ทำรอบถัดไปได้ ไม่บล็อกการปล่อย)

- **Server-side QR image endpoint** เช่น `GET /pay/:id/qr.png` คืน `image/png` จริง → ใช้เป็น fallback ที่นิ่งสุดสำหรับ WebView และทำให้ OG preview card โชว์ QR ได้ด้วย (เลิกพึ่ง canvas download ถาวร)
- ข้อความ reassurance หลังแชร์สำเร็จ ฝั่งแม่ค้า เช่น *"ส่งลิงก์ให้ลูกค้าแล้ว เมื่อลูกค้าจ่าย+ส่งสลิป ระบบจะตรวจให้อัตโนมัติ"* → ลดความกังวลว่าต้องทำอะไรต่อ
- Polish หน้า `EXPIRED` ให้ชัด

## 6. ❌ สิ่งที่ "ไม่ทำตอนนี้" (YAGNI)

ตัดออกชัด ๆ เพื่อไม่ให้รอบนี้บวม:

- **"Share Mode 3 แบบให้เลือก"** (แชร์ลิงก์ / ข้อความ+ลิงก์ / QR รูป) → ไม่ทำ default เป็น "แชร์ข้อความ+ลิงก์ก้อนเดียว" พอ
- **ปุ่มเยอะ ๆ บนหน้า success** (ดูรายละเอียด / สร้าง QR ใหม่ / กลับหน้าหลัก) → เหลือ "สร้างรายการใหม่" เป็นปุ่มหลัก + "กลับหน้าหลัก" เล็ก ๆ พอ
- **Websocket / realtime push** → polling เบา ๆ พอแล้วสำหรับ scale ตอนนี้ อย่า over-engineer
