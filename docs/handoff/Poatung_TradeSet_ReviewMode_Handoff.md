# แผนการปรับปรุง UX/UI — โหมด Trade Set (ชุดซื้อ-ขาย)
### Poatung · หมูนุ่น–เป๋าตุง · Engineering Handoff
**วันที่:** 8 มิถุนายน 2569
**โดย:** ตัดสินใจหลักจากการวิเคราะห์ + ประกอบข้อเสนอจาก 2 ทีมงาน
**สถานะ feature:** NLP / Auto-grouping ทำงานถูกต้องแล้ว (ต้นทุน 700 / ยอดขาย 2,400 / กำไร 1,700) — งานรอบนี้คือ **UX layer เท่านั้น ไม่แตะ parser**

---

## 0. สรุปสำหรับคนรีบ (TL;DR)

ปัญหาหลัก: หน้า "บันทึกรายการ" เอาโหมด **Input (กรอกเอง)** กับโหมด **Trade Set (AI สรุปให้)** มาซ้อนในหน้าเดียว → เกิด Cognitive Overload, numpad/category grid/ช่องแก้ไขซ้ำ แย่งความสนใจกับการ์ดสรุป

ทางแก้: เมื่อ `entryMode === "TRADE_SET"` ให้หน้าจอ **flip เป็น Review & Confirm Mode** — ซ่อน numpad + category grid, ดึงการ์ดสรุปขึ้นเป็นพระเอก, ย่อ wallet/date มาไว้ในการ์ด, ลบของซ้ำ

**3 จุดที่ต่างจากข้อเสนอ 2 ทีม และต้องทำตามนี้:**
1. **ความหนาแน่นของ UI ต้องผูกกับ confidence** (ไม่ใช่ collapse แบบเดียวทุกเคส) — กันเคส "confidently wrong"
2. **ซ่อน toggle รายจ่าย/รายรับ** ในโหมด Trade Set (มันขัดกับ concept "2 ทิศทาง")
3. **อย่าใช้ category เดียวกันทั้ง 2 leg แบบยัดรวม** — แยก expense category / income category ภายใน แต่แสดง business activity เดียวบนการ์ด (กัน reporting ภ.ง.ด. พัง)

---

## 1. หลักการตัดสินใจ (ทำไมถึงแก้แบบนี้)

ผู้ใช้ในโหมด Trade Set ไม่ได้กำลัง "กรอกยอดเดียว" แต่กำลัง **"ยืนยันชุดธุรกรรม 2 รายการ"** ที่ AI สรุปมาให้แล้ว งานของผู้ใช้คือ *ตรวจ → ปรับถ้าจำเป็น → บันทึก* ไม่ใช่ *พิมพ์ตัวเลข* ดังนั้น UI ต้องเปลี่ยน "verb" ของหน้าจอจาก **กรอก** เป็น **ยืนยัน**

ทุก element ที่ไม่ได้ช่วยให้ผู้ใช้ "ตัดสินใจยืนยัน" ได้เร็วขึ้น ต้องถูกซ่อนหรือย่อ

---

## 2. State machine กลาง (ฐานของงานทั้งหมด)

ตั้ง state เดียวคุมการ render ทั้งหน้า:

```
entryMode = "NORMAL" | "TRADE_SET" | "TRANSFER"
```

เมื่อ `entryMode === "TRADE_SET"`:

```
showToggleIncomeExpense = false   // ซ่อน toggle บนสุด
hideNumpad              = true
hideCategoryGrid        = true
showTradeSetCard        = true
showQuickEditActions    = true
removeBottomEditField   = true    // ลบช่องแก้ข้อความซ้ำ
```

> หมายเหตุ: รวม `"TRANSFER"` ไว้ใน enum ตั้งแต่ตอนนี้เพราะอนาคตจะมีเคสโอนเงินที่ logic คล้ายกัน (2 leg, ไม่นับเป็นกำไร) — เผื่อ pattern ไว้ ไม่ต้อง implement ตอนนี้

## 3. ⭐ Confidence-Tiered Rendering (จุดสำคัญที่สุด — เพิ่มจากข้อเสนอเดิม)

**ปัญหาของข้อเสนอเดิม:** ทั้ง 2 ทีมออกแบบบนสมมติฐาน "AI มั่นใจสูง" → collapse ทุกอย่างเหลือปุ่มเขียวปุ่มเดียว แต่ภาพทดสอบจริง badge ขึ้น **"มั่นใจปานกลาง"** ถ้าเรา collapse + ซ่อน edit path ในเคส medium/low ด้วย = เรากำลัง **เชียร์ให้ผู้ใช้กดยืนยัน parse ที่อาจผิด** ซึ่งคือ failure mode "confidently wrong" (เคส Toyota commission) ที่เราตั้งใจกันป้องกันมาตลอด

**กฎ:** ความหนาแน่น/ความเด่นของปุ่มยืนยัน ต้องแปรผกผันกับความเสี่ยงของ parse

| Confidence | การแสดงผล | ปุ่มยืนยัน | Edit affordance | Verify hint |
|---|---|---|---|---|
| **HIGH** (≥ threshold สูง) | Collapse เต็มที่ | เด่นเดี่ยว สีเขียวเต็ม | ปุ่ม "แก้ไขตัวเลข" เล็ก รอง | ไม่ต้องมี |
| **MEDIUM** (กลาง) | Review Mode แต่คง verify surface | เด่น แต่ **ไม่ dominant จนกดมั่ว** | "แก้ไขตัวเลข" + จุดที่ AI เดา (หมวด/ทิศ) เห็นชัด | แสดง hint สีเหลือง "ตรวจยอด/ทิศทางก่อนบันทึก" |
| **LOW** (ต่ำกว่า threshold) | **ไม่ collapse** | ไม่เด่น / require ยืนยันราย leg | บังคับให้ผู้ใช้ยืนยันต้นทุน + ยอดขายทีละตัว | แสดง warning "หมูนุ่นไม่แน่ใจ ช่วยตรวจหน่อยนะคะ" |

> Threshold ตัวเลขจริงให้ผูกกับ golden test dataset (P3) — อย่า hardcode ก่อนมีข้อมูล วาง enum `confidence: "HIGH" | "MEDIUM" | "LOW"` ที่ parser ส่งออกมาไว้ก่อน UI consume ตาม tier

mockup ที่แนบประกอบ handoff นี้คือเคส **MEDIUM** (badge เหลือง + hint + ปุ่มแก้ไขไม่ซ่อน)

## 4. ปรับราย Component (พร้อมเหตุผล)

### 4.1 Toggle รายจ่าย/รายรับ (บนสุด) — **ซ่อนในโหมด Trade Set**
**เหตุผล:** Trade Set เป็น **ทั้งรายจ่ายและรายรับ** การบังคับเลือกทิศเดียวขัด concept และ — สำคัญ — ในภาพ toggle ค้างที่ "รายรับ" คือ **ต้นเหตุ** ที่ทำให้ category grid เด้งมาเป็น income categories ล้วน (ทำให้ leg ต้นทุนไม่มีหมวด expense ให้เลือก ดูข้อ 4.3)
**ทำ:** ซ่อน toggle แทนด้วย pill อ่านอย่างเดียว `[ ⇄ ชุดซื้อ-ขาย ]` ที่มุมขวาบน

### 4.2 Numpad + ช่องยอด 2,400.00 — **ซ่อน, เปิดเมื่อกดแก้**
**เหตุผล:** AI คำนวณเสร็จแล้ว การโชว์ numpad ทำให้หน้าจอยาวเกินครึ่งและสื่อว่า "ยังกรอกไม่เสร็จ" → ผู้ใช้ลังเล
**ทำ:** ซ่อน numpad. การแก้ยอดทำผ่านปุ่ม "แก้ไขตัวเลข" → เปิด **Bottom Sheet** หรือ inline edit ให้แก้ ต้นทุน/ยอดขาย ได้ (ไม่ใช่เปิด numpad เต็มหน้า)
**ระวัง:** ในเคส MEDIUM/LOW ปุ่ม "แก้ไขตัวเลข" ต้องเข้าถึงง่าย ไม่ใช่ปุ่มจิ๋วที่มองไม่เห็น (ดูข้อ 3)

### 4.3 หมวดหมู่ — **ยุบ grid, แสดง business activity เดียว, แต่แยก namespace ภายใน** ⭐
**ปัญหาที่ข้อเสนอเดิมมองข้าม:** ทีม 1 เสนอ "ใช้หมวดเดียวกันทั้ง 2 leg" — แต่ทำไม่ได้จริงด้วย data model ปัจจุบัน เพราะ:
- category grid ในภาพเป็น **income namespace ทั้งแผง** (เงินเดือน, ดอกเบี้ย, ขายของในตลาด, ...)
- leg **ต้นทุนเป็น expense** ต้องใช้ expense category คนละ namespace — ในภาพไม่มีให้เลือกด้วยซ้ำ
- ถ้าฝืนยัดหมวดเดียว → ตอนทำรายงาน **ภ.ง.ด.90/91 / cash-basis** จะแยก "ต้นทุนขาย" ออกจาก "รายได้ขาย" ไม่ได้ → auditability พัง

**ทางแก้ (ตัดสินแล้ว):**
- บนการ์ด **แสดง label เดียว** = *business activity* เช่น "ขายของในตลาด" (สิ่งที่ผู้ใช้เห็นและเข้าใจง่าย)
- **ภายในเก็บแยก 2 field:**
  - cost leg → `expenseCategoryId` (เช่น `cost_of_goods` / "ต้นทุนสินค้า")
  - revenue leg → `incomeCategoryId` (เช่น `market_selling` / "ขายของในตลาด")
- ทั้งคู่ผูกกับ `businessActivity` เดียวกันเพื่อ group ในรายงาน
- **Auto-select + highlight** หมวดที่ AI เดา, ที่เหลือซ่อน. กด "เปลี่ยนหมวดหมู่ ▾" ค่อยเปิด
- ถ้าจะโชว์ตัวเลือก ให้โชว์เฉพาะ **3–4 หมวดที่เกี่ยวข้อง** (ขายของในตลาด / แม่ค้าออนไลน์ / ธุรกิจส่วนตัว) แล้วมี "ดูทั้งหมด"

> ผลข้างเคียงที่ดี: ทำแบบนี้แล้วรายงานสรุป "กิจการขายของในตลาด" จะดึงได้ทั้งต้นทุนและรายได้มา net กันเป็นกำไรต่อกิจกรรมได้เลย

### 4.4 กระเป๋าเงิน — **คงไว้ แต่ย่อเป็นบรรทัดในการ์ด, v1 ใช้กระเป๋าเดียว**
**เหตุผล:** จำเป็น (ต้องรู้เงินเข้า/ออกกระเป๋าไหน) แต่ปัจจุบันอยู่ล่างสุดโดน numpad ดันจนเข้าถึงยาก
**ทำ:**
- แสดงเป็นบรรทัดย่อในการ์ด: `กระเป๋าเงิน: เงินสด ▾`
- default = กระเป๋าหลัก / กระเป๋าที่ใช้ล่าสุด
- **v1 ใช้กระเป๋าเดียวทั้ง 2 leg** (เห็นด้วยกับทีม — YAGNI) ต้นทุนออก/ยอดขายเข้า กระเป๋าเดียวกัน
- Advanced split (ต้นทุนออกธนาคาร / รายรับเข้าเงินสด) → เลื่อนไป backlog

### 4.5 ปฏิทิน — **คงไว้ ย่อเป็นบรรทัด, default = วันนี้**
**เหตุผล:** จำเป็นสำหรับบันทึกย้อนหลัง (ขายเมื่อวาน เพิ่งมาพิมพ์วันนี้)
**ทำ:** บรรทัดย่อ `วันที่: 8 มิ.ย. 2569 ▾` ไม่แตะ = ใช้วันนี้. v1 ใช้วันที่เดียวทั้งชุด (แยกวันซื้อ/วันขาย → backlog)

### 4.6 ช่องแก้ไขข้อความด้านล่าง (ดินสอ "ซื้อลูกชิ้น 700...") — **ลบทิ้ง**
**เหตุผล:** ซ้ำซ้อน 100% กับช่อง input บนสุด การแก้ข้อความ source ทำที่ช่องบนได้อยู่แล้ว

### 4.7 การ์ดไกด์/คำอธิบายล่างสุด (รูปกุญแจ) — **ย่อ + ย้ายไปใต้ปุ่มบันทึก**
**เหตุผล:** จำเป็นในฐานะ guide แต่อยู่ไกลเกินไป ผู้ใช้ต้องเลื่อนหา
**ทำ:** ย่อเหลือ 1 บรรทัดใต้ปุ่มเขียว เช่น *"ระบบจะบันทึก 2 รายการพร้อมกัน: ต้นทุน และ ยอดขาย"*

## 6. Data model

### Draft (ก่อนบันทึก)
```js
tradeSetDraft = {
  type: "BUY_SELL",
  confidence: "MEDIUM",            // ⭐ HIGH | MEDIUM | LOW — คุม UI tier
  costAmount: 700,
  revenueAmount: 2400,
  profit: 1700,                    // คำนวณ ไม่เก็บแยก = revenue - cost
  businessActivity: "market_selling",
  expenseCategoryId: "cost_of_goods",   // ⭐ leg ต้นทุน (expense namespace)
  incomeCategoryId: "market_selling",   // ⭐ leg ขาย (income namespace)
  walletId: "cash",               // v1 กระเป๋าเดียว
  date: "2026-06-08",
  sourceText: "ซื้อลูกชิ้น 700 บาทไปทอดขายในตลาดได้ 2400 บาท"
}
```

### ตอนบันทึก → สร้าง 2 transaction ผูกด้วย `tradeSetId`
```js
const tradeSetId = generateId();
[
  {
    type: "expense",
    title: "ต้นทุน: ซื้อลูกชิ้น",
    amount: 700,
    categoryId: "cost_of_goods",
    businessActivity: "market_selling",
    walletId: "cash",
    date: "2026-06-08",
    tradeSetId,                    // ⭐ ผูกชุด
    tradeRole: "COST"
  },
  {
    type: "income",
    title: "ยอดขาย: ขายลูกชิ้นในตลาด",
    amount: 2400,
    categoryId: "market_selling",
    businessActivity: "market_selling",
    walletId: "cash",
    date: "2026-06-08",
    tradeSetId,
    tradeRole: "REVENUE"
  }
]
```

### History
ใช้ `tradeSetId` group 2 transaction แสดงเป็น card เดียว:
```
ชุดซื้อ-ขาย: ลูกชิ้น
ต้นทุน 700 · ยอดขาย 2,400 · กำไร 1,700
```
แตะเข้าไปดู 2 รายการย่อยได้ (ตรงนี้ทำได้แล้วตามที่ทดสอบ — แค่ให้แน่ใจว่า group key ใช้ `tradeSetId`)

## 7. ลำดับความสำคัญ (Priority)

**P1 — ทำก่อน (แก้ overload ทันที):**
- State machine `entryMode` + flip เป็น Review Mode
- ซ่อน numpad + category grid + toggle รายจ่าย/รายรับ
- ลบช่องแก้ข้อความซ้ำ (4.6), ย้าย guide ขึ้น (4.7)
- ดึง wallet/date มาเป็นบรรทัดในการ์ด

**P2 — ตามมา (ความถูกต้อง + ความสวย):**
- แยก `expenseCategoryId` / `incomeCategoryId` + business activity (4.3)
- Auto-select + highlight หมวดที่ AI เดา
- Bottom sheet "แก้ไขตัวเลข"
- ปุ่ม "แยกเป็นรายการปกติ"

**P3 — ผูกกับข้อมูล (อย่ารีบ):**
- Confidence-tiered rendering แบบเต็ม (ผูก threshold กับ golden test dataset)
- LOW confidence flow (ยืนยันราย leg)

## 8. ข้อตัดสินใจที่ค้าง ให้ทีมเคาะในที่ประชุม

1. **Threshold ของ confidence tier** — รอ golden test dataset (P3) อย่า hardcode ก่อน
2. **`cost_of_goods` เป็น category ใหม่หรือ map กับหมวด expense ที่มีอยู่?** — ต้องเช็ค expense category ปัจจุบันก่อน
3. **เคส LOW confidence** — ยืนยันราย leg แบบไหนไม่ให้รำคาญ? (เสนอ: inline confirm 2 บรรทัด ไม่ใช่ popup ซ้อน)
4. **"แยกเป็นรายการปกติ"** — แตกเป็น 2 transaction อิสระ (ตัด tradeSetId) หรือคงไว้แค่เปลี่ยน view? (เสนอ: ตัด tradeSetId จริง เพื่อความตรงไปตรงมา)

## 9. สรุป

ระบบ "ฉลาดแล้ว" — รอบนี้คือทำให้หน้าจอ **นิ่ง สะอาด มั่นใจ** เหมือนผู้ช่วยการเงินมืออาชีพ ไม่ใช่ฟอร์มที่เปิดทุกอย่างพร้อมกัน

หัวใจ 3 ข้อที่ห้ามพลาด:
1. **UI density ผูกกับ confidence** (อย่า collapse แบบเดียวทุกเคส)
2. **ซ่อน toggle รายจ่าย/รายรับ** ในโหมดนี้
3. **แยก category namespace ภายใน** แต่แสดง activity เดียวบนการ์ด (กัน ภ.ง.ด. reporting พัง)

ที่เหลือ (ซ่อน numpad/grid, ย่อ wallet/date, ลบของซ้ำ, ผูก tradeSetId) = ตรงตามข้อเสนอ 2 ทีม ทำได้เลย
